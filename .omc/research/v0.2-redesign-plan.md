# claudit v0.2 — Redesign Plan

## Goal

Make claudit actually detect its own flagship scenario (rtk + OMC), and replace v0.1's assumption-driven snapshot+detectors with evidence-driven ones. v0.2 keeps the Scanner/Report/hook-IO/pending architecture (they were fine); it rewrites `snapshot.ts` and substantially refactors 4 of 6 detectors.

## Preserved Architecture (no changes)

- `src/types.ts` Collision/Report schema (add new fields, don't break existing)
- `src/detector.ts` async interface
- `src/scanner.ts` Promise.allSettled + per-detector timeout
- `src/report.ts` serialize/parse, with XML-escape hardening
- `src/hook-io.ts` stdin parse / stdout build
- `src/pending.ts` atomic write + hash-suffix filenames (add redaction)
- `src/policies.ts` constants (consolidated in v0.1.1-scope)
- `src/hooks/post-tool-use.ts` trigger+defer (add redaction)
- `src/hooks/session-start.ts` consume markers, diff, run scanner

## Rewritten Modules

### `src/snapshot.ts` — full rewrite

Replace v0.1 snapshot with:

1. **`HookSource`** enumeration: `plugin-cache`, `plugin-marketplace`, `user-settings`, `user-settings-local`, `project-settings`, `project-settings-local`.
2. **`capturePluginsFromCache()`** — walk `~/.claude/plugins/cache/<marketplace>/<plugin>/<version>/` exactly 3 levels deep. Plugin root contains `.claude-plugin/plugin.json`.
3. **`capturePluginsFromMarketplaces()`** — walk `~/.claude/plugins/marketplaces/<marketplace>/` and `.../marketplaces/<marketplace>/plugins/<plugin>/`. Treat marketplace root itself as a plugin root when it has hooks/skills/agents/commands directly.
4. **`captureUserSettings()`** — read `~/.claude/settings.json` AND `~/.claude/settings.local.json`. Extract `hooks`, `mcpServers` (if present), `enabledPlugins`.
5. **`captureProjectSettings()`** — read `<cwd>/.claude/settings.json` AND `<cwd>/.claude/settings.local.json` (when run from a project root).
6. **Replace `parseMdFrontmatter`** with `js-yaml` or a focused YAML subset parser (block lists + inline lists + multiline strings). Add `js-yaml` to devDependencies is cheap; we can vendor a small subset-parser instead to keep runtime deps at zero.
7. **Normalize `plugin.json.skills`**: handle `string (dir path) | array of paths | absent`. When dir path, walk for `SKILL.md`. When array, treat each entry as a skill dir or file.
8. **Normalize `plugin.json.mcpServers`**: handle `string (path to .mcp.json) | inline object | absent`. When string path, resolve relative to plugin root and read `.mcp.json`.
9. **Filter by `enabledPlugins`**: a plugin's scan-status is tagged `enabled | disabled | unknown`.

### Detectors

#### `hook-matcher.ts` — major change

- Accept hooks from all 5 sources (not just plugins).
- Add source metadata to each hook entry: `{source: HookSource, sourceId: string, event, matcher, command, scriptSource}`.
- Replace string-equality grouping with **matcher overlap**:
  - Compile `"*"`, single tool name, pipe-OR list `"A|B"` into sets.
  - Two hooks interfere when their matcher sets intersect AND their event names match.
- Keep the mutation classifier but drop `JSON.stringify` as a standalone signal (require assignment/object-emission evidence).

#### `slash-command.ts`, `skill-name.ts`, `subagent-type.ts` — semantics fix

- Same base name in two ENABLED plugins is no longer `definite`/destructive. Downgrade to `severity: info, confidence: possible`.
- Message: "ambiguous when invoked without prefix — prefer `plugin:name` to disambiguate".
- Remove "destructive uninstall" fix suggestions. Replace with "edit the manifest" manual-review fixes that are actually executable (or genuinely non-actionable and labeled so).
- Trigger-keyword overlap in skill-name stays `possible` (already correct).

#### `mcp-identifier.ts` — extend input

- Read user-level `.mcp.json` (try `~/.claude/.mcp.json`, `~/.mcp.json`) and project-level `.mcp.json` in addition to plugin-declared servers.
- Keep existing definite (server name) / possible (tool name) logic.

#### `path-binary.ts` — small fix

- `capturePathBinaries` must filter to regular executable files only: `stat.isFile() && (mode & 0o111) !== 0`.
- `safeHash` follows symlinks (use `stat` not `lstat`) with realpath-based loop/device guard; hash the target. Skip only when target is non-regular.
- Allowlist unchanged.

### `src/report.ts` — XML hardening

Replace naive wrapper with:
- Base64 the JSON payload, or
- Escape literal `</claudit-report>` occurrences inside the payload (unlikely but possible via plugin names/paths).

Go with base64 — cleaner and fully escape-proof. Report parser decodes on the way back. Breaks the "human-readable raw output" property but the scan.md prompt will tell Claude to base64-decode.

### `src/hooks/post-tool-use.ts` — redaction

Before writing the pending marker, apply a redactor to the captured command:
- Replace bearer tokens in URLs: `https://…?token=XXX` → `https://…?token=<redacted>`.
- Replace `Authorization: Bearer …` patterns (via curl `-H`).
- Replace env-var assignment values in install lines (`FOO=secret brew install` → `FOO=<redacted> brew install`).
- Redacted command appears in both pending marker and `additionalContext`.

### `src/detectors/*` — FixSuggestion cleanup

- Purge every `# comment` pseudo-command. Replace with one of:
  - An actually runnable command (e.g., `/plugin disable plugin-b` when we know it's safe), OR
  - An empty command string with `safety_level: "manual-review"` and a message-only rationale. Scan.md must know to treat empty command as "tell the user, don't execute".

### `commands/scan.md` — prompt hardening

Add explicit directives:

1. "Execute `node $CLAUDE_PLUGIN_ROOT/dist/commands/scan.mjs` and capture stdout."
2. "Find the `<claudit-report>…</claudit-report>` block. Extract the base64 payload, decode it, parse the JSON. **Do not show raw stdout to the user** — summarize only."
3. For each collision: translate `message` into plain English, show severity+confidence as a user-facing badge.
4. For `possible` or `unknown` confidence: **do not offer a fix**, ask the user to inspect.
5. For `definite` with `safety_level: "safe"` fixes: offer to execute, wait for yes.
6. For `safety_level: "destructive"` or `"manual-review"`: explain, show the command as text only, DO NOT execute.
7. For FixSuggestion with empty `command`: display the rationale as guidance; do not invoke Bash.

### Error-message polish

- Replace `updatedInput` jargon with plain language: "both plugins try to modify the same Bash command; only one will win".
- Run all Collision.message strings through an "explain-like-to-a-user" pass.

## Scope boundaries

- Stay zero-runtime-dependency if possible. Vendor a small YAML subset parser instead of adding `js-yaml`. If that becomes too hard, accept `js-yaml` as the only runtime dep.
- Don't add fs.watch / background mode. Keep trigger+defer architecture.
- Don't introduce SQLite for pending markers. File-per-event is fine at our volume.
- Don't attempt runtime sandboxing of hook scripts (Gemini's suggestion). Out of scope for v0.2.

## Open Questions for ralplan (answer before execution)

1. YAML parser: vendor subset parser (zero deps) vs `js-yaml` dev-dep-only vs `js-yaml` runtime-dep.
2. Base64 report wrapper vs literal-escape: base64 is cleaner but visible stdout becomes opaque. Scan.md must handle both forms if we pick base64.
3. Pending marker redaction policy: redact aggressively (any query string value) vs conservatively (only known secret patterns).
4. Namespace detector: should "same name in two plugins where the user often types `/name` without prefix" still warn? Or is the namespace guarantee strong enough to skip entirely?
5. `enabledPlugins` handling: disabled plugins excluded entirely, or included with collisions marked `confidence: possible` because "user might re-enable later"?
