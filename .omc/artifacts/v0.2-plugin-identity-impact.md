# v0.2 plugin identity impact map

Date: 2026-04-20
Scope: marketplace-qualified `enabledPlugins` resolution (`plugin@marketplace`) and every current use of plugin identity that can be affected by name-only normalization.

## Ground truth

- `enabledPlugins` keys are marketplace-qualified, e.g. `codex@openai-codex`, `oh-my-claudecode@omc`.
- Source: `.omc/research/cc-schema-ground-truth.md:110-120`

## Snapshot fields / normalization points

### PluginSummary fields carrying identity today
- `src/types.ts:153-163`
  - `PluginSummary.name`
  - `PluginSummary.pluginRoot`
  - `PluginSummary.enabled`
- There is currently **no dedicated marketplace-qualified plugin identity field**.

### Snapshot capture / normalization points
- `src/snapshot.ts:439-474`
  - `capturePlugin()` sets only `name` from `plugin.json.name` (or folder fallback).
- `src/snapshot.ts:157-160`
  - `enabledPlugins` maps are merged from user + project settings.
- `src/snapshot.ts:603-607`
  - `enabledPlugins` entries are ingested verbatim from settings JSON.
- `src/snapshot.ts:1117-1135`
  - `applyEnabledPluginsFilter()` and `resolveEnabledPluginState()` currently normalize by bare plugin name and can collapse `foo@alpha` / `foo@beta`.
- `src/snapshot.ts:243-244`
  - Snapshot diff keys plugins by `name`, not by a marketplace-qualified identity.

## Detectors using plugin identity

### 1) slash-command
- `src/detectors/slash-command.ts:16-42`
- Uses `plugin.name` for:
  - grouping (`byName` entries hold `{ plugin: plugin.name, enabled: plugin.enabled }`)
  - de-duplication (`pluginMap` keyed by plugin name)
  - emitted entities (`${plugin}:/${command}`)
- Impact:
  - same plugin name across marketplaces can collapse into one logical plugin
  - enabled/disabled state can be merged incorrectly

### 2) skill-name
- `src/detectors/skill-name.ts:19-69`
- Uses `plugin.name` for:
  - name collision grouping
  - trigger overlap origin strings
  - de-duplication and emitted entities
- Impact:
  - same-name different-marketplace skills can collapse into one origin
  - disabled state may be inherited from the wrong marketplace-qualified setting

### 3) subagent-type
- `src/detectors/subagent-type.ts:14-42`
- Uses `plugin.name` for grouping, de-duplication, and emitted entities.
- Impact:
  - same-name different-marketplace agents can collapse into one plugin origin

### 4) mcp-identifier
- `src/detectors/mcp-identifier.ts:86-102`
- Uses `plugin.name` as MCP server `origin` for plugin-declared servers.
- Impact:
  - MCP findings from `foo@alpha` and `foo@beta` are indistinguishable
  - enabled-state decisions upstream can mislabel plugin MCP findings

### 5) hook-matcher
- `src/detectors/hook-matcher.ts:214-215`
- Plugin hook entity string is `${plugin.name}:${event}:${matcher}`.
- Impact:
  - same-name different-marketplace plugins become ambiguous in emitted entities
  - disabled/active state is sourced from `plugin.enabled`, so upstream resolution bug can change confidence/severity

## Affected behavior summary

1. **Enablement resolution** can be wrong when two marketplaces publish the same plugin name.
2. **Detector dedupe/grouping** can collapse distinct installs into one origin because they key on `plugin.name`.
3. **Entity strings/messages** can become ambiguous because they do not carry a marketplace-qualified origin.
4. **Snapshot diffing** currently keys by bare plugin name and can treat marketplace-identity changes as the same plugin.

## Minimal-intrusion fix direction

- Introduce a **single centralized plugin identity utility** that derives:
  - display name (`plugin.name`)
  - marketplace-qualified key (`plugin.name@marketplace` when marketplace known)
- Store the qualified key on `PluginSummary` once during snapshot capture.
- Reuse that one field/helper for:
  - `enabledPlugins` lookup
  - detector origin/de-duplication keys
  - emitted entity strings where uniqueness matters
- Preserve existing user-facing display-name semantics where possible, but stop using bare `plugin.name` as the only internal identity.
